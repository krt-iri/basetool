### Added
- **Docs (Klassen- und Methoden-Javadoc fuer alle Backend-Repositories, 202 Checkstyle-Verstoesse weggeraeumt)**: Phase 2c der Checkstyle-Aufraeumaktion. Behandelt das gesamte `backend.repository`-Paket: 41 Klassen-Javadocs auf allen Spring-Data-Interfaces und 161 Methoden-Javadocs auf allen derived queries und Custom-`@Query`-Methoden. Klassen-Javadocs kamen ueber das erweiterte Hilfs-Skript (`build/_javadoc_insert.py`) mit neuer Suffix-Regel `*Repository` → `Spring Data repository for X`. Methoden-Javadocs wurden ueber ein zweites, dediziertes Hilfs-Skript (`build/_javadoc_methods_insert.py`) generiert, das die Spring-Data-Method-Name-Praefixe parst und passende 1-Zeiler emittiert: `findBy*` → "Derived Spring-Data query - returns entities matching X", `existsBy*` → "Returns {@code true} iff at least one row matches X", `countBy*` → "Returns the count of rows matching X", `deleteBy*` → "Deletes every row matching X", `findFirstBy*`/`findTopBy*` → "Returns the first matching X (limit 1)", `findAll*` → "Lists every entity. Overridden here to attach an @EntityGraph". Annotation-Detection ergaenzt den 1-Zeiler: `@EntityGraph` haengt "Eagerly fetches the configured relations via @EntityGraph" an, `@Lock` haengt "Acquires a pessimistic write lock" an. **Wichtige Qualitaets-Einschraenkung:** Custom-`@Query`-Methoden (~40 Stellen, vor allem in `InventoryItemRepository`, `MissionRepository`, `MaterialPriceRepository`) bekommen einen generischen 1-Zeiler ("Custom JPQL/native query; see the {@code @Query} annotation for the projection and filter clauses"). Das erfuellt den Checkstyle-Check, ist aber Boilerplate und sollte in einem dedizierten Refinement-Pass durch inhaltlich aussagekraeftige Javadocs ersetzt werden. Verifiziert per `./gradlew :backend:test` (BUILD SUCCESSFUL in 54s, alle 1300+ Backend-Tests gruen).

### Fixed
- **Backend (`OperationService.getOperationPayouts` N+1-Lazy-Load eliminiert)**: Die Payout-Berechnung iterierte ueber `operation.getMissions()` (lazy `OneToMany`), dann pro Mission ueber `mission.getParticipants()` (lazy `OneToMany`) — fuer N Missions mit je M Participants ergab das `1 + N + (N*M)` Roundtrips zur DB. Bei einer Operation mit 5 Missions und je 10 Teilnehmern waren das 56 Queries fuer einen einzigen Endpoint-Call. `MissionParticipant.user` ist zwar `@ManyToOne` (per JPA-Default EAGER), das passierte also implizit im selben Participant-Select; der eigentliche Wasserfall waren die Mission- und Participant-Collection-Loads. Fix: neue Repository-Methode `OperationRepository.findWithMissionsAndParticipantsById(UUID)` mit `@EntityGraph(attributePaths = {"missions", "missions.participants", "missions.participants.user"})` — Hibernate emittiert das in einer einzelnen Query mit LEFT-JOINs auf alle drei Ebenen. Der `user`-Path ist redundant (EAGER ManyToOne kommt eh mit), aber explizit ist hier besser als implizit. `OperationService.getOperationPayouts` ruft jetzt diese Methode statt des Plain-`findById` auf; die Iteration darunter bleibt unveraendert, jeder Aufruf von `.getMissions()` / `.getParticipants()` / `.getUser()` trifft den schon initialisierten Cache. Tests entsprechend angepasst: `GetOperationPayoutsTests.stubOperation`-Helper und `throwsNotFound_whenOperationDoesNotExist` mocken jetzt `findWithMissionsAndParticipantsById` statt `findById`. Das macht die Tests gleichzeitig zu einer indirekten Regressions-Sicherung — wuerde jemand den Service zurueck auf `findById` umstellen, faellt der Stub durchs Raster und die 24 Payout-Tests scheitern an NPE/NotFound. Volle Backend-Suite bleibt gruen.

### Added
- **Backend (`Operation`-Lifecycle-State-Machine inkl. ADMIN-Bypass)**: Der `OperationController.updateOperation`-Endpoint akzeptierte bislang jeden beliebigen Status-Wechsel — `COMPLETED → ACTIVE → PLANNED` war genauso möglich wie alles andere. Fachlich gewollt ist eine klare Vorwärts-Pipeline: `PLANNED → {ACTIVE, CANCELED}`, `ACTIVE → {COMPLETED, CANCELED}`, `COMPLETED` und `CANCELED` sind terminal. Same-status-Updates (z. B. Description-Änderung auf einer COMPLETED-Operation) müssen erlaubt bleiben, weil sonst die Edit-API für abgeschlossene Operationen wertlos wäre. Fix: (1) Neue Methode `OperationStatus.canTransitionTo(OperationStatus)` als Switch-Expression direkt im Enum — Single Source of Truth fuer die State Machine. Same-Status liefert immer `true`; sonst Forward-Path-Matrix. (2) `OperationService.updateOperation(UUID, OperationUpdateDto, boolean canOverrideStatus)` bekommt einen dritten Boolean-Parameter, der den State-Gate-Check kurzschließt. Bei `canOverrideStatus=false` und einer unerlaubten Transition fliegt eine `BadRequestException` (RFC-7807 `400` mit Code `BAD_REQUEST` und `detail` "Invalid operation status transition: PLANNED -> COMPLETED"), bei `canOverrideStatus=true` wird der Check skipped. (3) `OperationController.updateOperation` injiziert das Spring-Security-`Authentication`-Objekt am HTTP-Boundary, liest die GrantedAuthorities, setzt `canOverrideStatus=true` wenn `ROLE_ADMIN` dabei ist — exakt der Pattern, der schon in `HangarController.getSquadronOverview` verwendet wird (kein `SecurityContextHolder` im Service, Architektur-Regel ist via ArchUnit-Test erzwungen). (4) Neuer fokussierter Unit-Test `OperationStatusTest` (3 Tests, 12 parametrisierte Cases) pinnt die State-Machine direkt am Enum — getrennt von den Service-Flow-Tests, damit ein Loosening hier einen sofortigen Test-Fail produziert statt sich durch die Service-Layer-Tests zu schleichen. (5) Vier neue Tests in `OperationServiceTest.UpdateOperationTests`: `rejectsForbiddenStatusTransition_whenNotAdmin` (PLANNED → COMPLETED → 400), `terminalStatusCannotBeChanged_whenNotAdmin` (COMPLETED → ACTIVE → 400), `sameStatusIsAlwaysAllowed_evenWithoutAdmin` (COMPLETED → COMPLETED nur mit neuem Namen geht durch — Edit-API muss auf terminalen Operationen weiterhin funktionieren), `adminMayOverrideStatusMachine` (CANCELED → PLANNED mit `canOverrideStatus=true`). Die bestehenden `updatesAllFields_*`/`throwsOptimisticLockingFailure_*`/`acceptsNullVersionInIncoming_*` Tests entsprechend auf gueltige Transitions umgestellt (z. B. PLANNED → ACTIVE statt PLANNED → COMPLETED).

### Changed
- **Backend (`OperationService.updateOperation` nimmt jetzt die `OperationUpdateDto` direkt entgegen, `OperationMapper.updateEntity` entfernt)**: Bisheriger Flow im Controller war eine fragile Choreographie: (1) `operationService.getOperationById(id)` lieferte die Entity zurück, (2) `operationMapper.updateEntity(updateDto, operation)` patchte die DTO-Felder via MapStruct in die Entity, (3) der Service rief erneut `findById` auf und prüfte die Version. Problem: Der Mapper hatte `unmappedTargetPolicy = ReportingPolicy.IGNORE` und keinen expliziten `@Mapping(target = "version", ignore = true)` — MapStruct mappte deshalb implizit auch die `version` der DTO auf die geladene (potenziell gemanagte) Entity. Der nachfolgende manuelle Versions-Check im Service nutzte diese mapper-übertragene Version. Das funktionierte zufällig richtig, aber war komplett unsichtbar im Code und durch keinen Test pinned — eine MapStruct-Refaktorisierung, ein neues Mapping-Profil oder das Hinzufügen weiterer `@Mapping(target = X, ignore = true)`-Direktiven (mit Copy-Paste-Fehler) hätte die Optimistic-Locking-Garantie still gekippt. Fix: das Pattern an `JobOrderService.updateJobOrderStatus` angeglichen, das im Projekt schon seit langem das Vorbild ist. (1) `OperationService.updateOperation(UUID, Operation)` umgestellt auf `updateOperation(UUID, OperationUpdateDto)` — Service liest die Versions-Erwartung direkt aus `updateDto.version()`, kein Umweg mehr über eine gemanagte Entity. (2) `OperationMapper.updateEntity(OperationUpdateDto, @MappingTarget Operation)` ersatzlos gestrichen — keine Call-Site mehr, weil der Service die DTO-Felder per Setter direkt anwendet. Mapper-Interface enthält nur noch `toDto(Operation)` für Outbound und `toEntity(OperationCreateDto)` für Create. (3) `OperationController.updateOperation` reduziert sich auf eine einzige Zeile: `return operationMapper.toDto(operationService.updateOperation(id, updateDto));` — die doppelte `getOperationById`+`mapper.updateEntity`-Choreographie ist weg. Tests entsprechend angepasst: `OperationServiceTest.UpdateOperationTests` (4 Tests) baut die incoming-Daten jetzt als `OperationUpdateDto` statt als nackte `Operation`-Instanz, der `acceptsNullVersionInIncoming_asBypassToken`-Test dokumentiert explizit, dass der Bypass-Pfad an der DTO-Boundary nur noch durch Service-interne Aufrufer erreichbar ist (Controller-Boundary erzwingt `@NotNull` auf `version`). `OperationMapperTest` verliert den `updateEntity_shouldOverwriteScalarFieldsButNotIdTimestampsOrMissions`-Test (Methode existiert nicht mehr).
