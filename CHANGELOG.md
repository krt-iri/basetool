# Changelog
## [Unreleased]

### Added
- **Docker (HEALTHCHECK fuer Backend und Frontend)**: Beide Dockerfiles erhalten eine `HEALTHCHECK`-Direktive, die den Spring-Boot-Actuator-Health-Endpoint anpingt (`https://localhost:11261/actuator/health` bzw. `https://localhost:18081/actuator/health`, jeweils mit `curl -k`, da die Container HTTPS mit Self-Signed-Cert servieren). Damit kann `docker compose` Container per `condition: service_healthy` orchestrieren; das `frontend-(dev)`-Service nutzt das jetzt fuer `backend`/`backend-dev` (vorher `condition: service_started`). Konkret: (1) `backend/src/main/resources/application.yml` ergaenzt einen `management:`-Block, der `endpoints.web.exposure.include=health` setzt und `endpoint.health.show-details=never`/`show-components=never` erzwingt — die Antwort ist auf `{"status":"UP"}` reduziert, sodass unauthentifizierte Clients keine internen Komponenten enumerieren koennen; `endpoint.health.probes.enabled=true` aktiviert die Liveness/Readiness-Sub-Endpoints fuer Kubernetes-aehnliche Umgebungen. (2) `backend/.../SecurityConfig.java` `permitAll`-t `/actuator/health` und `/actuator/health/**`; alle anderen Actuator-Endpoints fallen weiterhin unter `anyRequest().authenticated()`. (3) Frontend bekommt erstmals `org.springframework.boot:spring-boot-starter-actuator` als Dependency (siehe `frontend/build.gradle.kts`); analoger `management:`-Block plus `permitAll` im SecurityConfig. (4) `BotProtectionFilter` blockte bisher `/actuator/*` komplett als Scanner-Pfad; eine neue `LEGITIMATE_PATHS`-Whitelist (aktuell nur `/actuator/health`) wird in `isBotPath(...)` _vor_ der Praefix-Pruefung konsultiert, sodass `/actuator/health[/...]` durchgereicht wird, alle anderen `/actuator/...`-Pfade aber weiterhin als Bot-Pfad mit HTTP 404 abgewiesen werden. Korrespondierende Updates in `BotProtectionFilterTest` (Whitelist-Verhalten als eigener Test, das alte `/actuator/health`-Bot-Path-Assertion-Beispiel durch `/actuator/env` ersetzt). (5) Beide Dockerfiles installieren `curl` + `ca-certificates` im finalen Image (Eclipse-Temurin-JRE-Jammy enthaelt es nicht standardmaessig, etwa +5 MB pro Image) und konfigurieren `HEALTHCHECK --interval=30s --timeout=5s --start-period=60s --retries=3 CMD curl -fsSk <url>/actuator/health || exit 1`. Die `start-period=60s` deckt JVM-Warm-up + Tomcat-Bind + Flyway ab, sodass Compose den Container waehrend des Boot-Vorgangs nicht in `unhealthy` flippt. Frontend-SBOM (`frontend/docs/frontend-bom.{json,xml}`) regeneriert mit der neuen Actuator-Dependency.

### Changed
- **Build (gemeinsame Test/JavaCompile/BootRun-Konfiguration zentralisiert)**: `tasks.withType<Test>`, `tasks.withType<JavaCompile>` und `tasks.withType<org.springframework.boot.gradle.tasks.run.BootRun>` waren bisher in `backend/build.gradle.kts` und `frontend/build.gradle.kts` praktisch identisch dupliziert (JUnit-Platform, native-access-JVM-Arg, Spring-Profile, Mockito-Java-Agent-Anhängen, UTF-8-Encoding, `-Xlint:unchecked`/`-Xlint:deprecation`). Die gemeinsame Konfiguration wandert in einen `subprojects { plugins.withId("java") { ... } plugins.withId("org.springframework.boot") { ... } }`-Block im Root-`build.gradle.kts` — DRY und reagiert nur dort, wo das Java- bzw. Spring-Boot-Plugin tatsächlich angewendet ist. Frontend behält die JaCoCo-Erweiterung (`tasks.withType<Test>().configureEach { finalizedBy(tasks.jacocoTestReport) }`) als modulspezifischen Zusatz. Beiläufig vereinheitlicht: Backend bekommt jetzt ebenfalls den `-parameters`-Compiler-Flag (war vorher nur Frontend), was Spring/Jackson-Reflection auf Method-Parameter-Namen zuverlässiger macht und CLAUDE.md-Konventionen folgt. BootRun-Konfiguration nutzt `tasks.named<JavaExec>("bootRun")` (BootRun erbt von JavaExec), damit der Root-Build keine direkte Abhängigkeit zur Spring-Boot-Plugin-Klasse braucht. Verifiziert via `./gradlew :{backend,frontend}:compileJava --debug` (zeigt `-parameters`, `-Xlint:*`) und Test-Runner-Kommandozeile (zeigt `-javaagent:.../mockito-core-5.20.0.jar` und `-Dspring.profiles.active=test` für beide Module).

### Fixed
- **Backend-Dockerfile (`EXPOSE`-Port korrigiert)**: `backend/Dockerfile` exponierte als Image-Metadatum den Port `10261`, obwohl der Spring-Boot-Container tatsächlich auf `11261` lauscht (siehe `application.yml:8` und das `SERVER_PORT: 11261` im `docker-compose.yml:70`). Der Tippfehler hatte keinen Funktionseffekt — `EXPOSE` ist in Docker reines Metadatum, der Port-Bind passiert über `-p`/Compose — verwirrte aber Tools wie `docker inspect`, IDE-Container-Plugins und Healthcheck-Generatoren. Auf `EXPOSE 11261` korrigiert.
- **Build (JetBrains-Annotation-Version zentralisiert)**: `backend/build.gradle.kts` und `frontend/build.gradle.kts` deklarierten `compileOnly("org.jetbrains:annotations:…")` mit jeweils hartcodierter, divergenter Version (`26.1.0` vs `26.0.2`), was sowohl der CLAUDE.md-Regel widersprach ("Dependencies are managed by refreshVersions — edit `versions.properties`, not `build.gradle.kts`") als auch zur Inkonsistenz führte. Die Version wandert in `versions.properties` als `version.org.jetbrains..annotations=26.1.0`, beide Module nutzen jetzt das refreshVersions-Sentinel `compileOnly("org.jetbrains:annotations:_")`, und damit ist die Version automatisch identisch und kann per `./gradlew refreshVersions` auf Updates geprüft werden.

Aufruf via `./gradlew :frontend:cyclonedxBom`. Hintergrund: laut Analyse 2.6 fehlte dem Frontend bislang die Supply-Chain-Transparenz, obwohl es produktiv ausgerollt wird und über sicherheitsrelevante Dependencies wie Spring Boot 4 (Web/Security/OAuth2-Client/Session/Redis), Resilience4j, Logstash-Encoder, Thymeleaf, Caffeine, Jackson und MockWebServer verfügt. Konkret: Plugin `alias(libs.plugins.cyclonedx.bom)` im `plugins {}`-Block aktiviert (Plugin-Definition im bestehenden Version-Katalog wird mitbenutzt, keine neue Version-Eintragung nötig); `tasks.cyclonedxBom { ... }`-Block kopiert die Konfiguration aus `backend/build.gradle.kts` 1:1 (Schema v1.6, Output unter `docs/`, `includeBomSerialNumber`, `includeLicenseText`, `includeBuildSystem`). `CLAUDE.md` listet den neuen Befehl unter "Build, run, test" auf. - **Backend (Domänen-Exceptions ersetzen `ResponseStatusException`)**: Zwei neue Exception-Typen, `BadRequestException` (HTTP 400) und `BusinessConflictException` (HTTP 409), ergänzen die bestehenden `NotFoundException`/`DuplicateEntityException`/`EntityInUseException`. Damit verschwindet `ResponseStatusException` aus dem Service-Layer-Throw-Pfad: alle 85 Aufrufe in 14 Production-Dateien (Controller `JobOrderController`/`MissionController` sowie Services `FrequencyTypeService`, `HangarImportService`, `HangarService`, `InventoryItemService`, `JobOrderHandoverReportService`, `JobOrderHandoverService`, `JobOrderService`, `MaterialCategoryService`, `MissionFinanceEntryService`, `MissionSecurityService`, `MissionService`, `SystemSettingService`) wurden auf den passenden Domänentyp umgestellt: `HttpStatus.NOT_FOUND` → `NotFoundException`, `HttpStatus.BAD_REQUEST` → `BadRequestException`, `HttpStatus.CONFLICT` (Ambiguität) → `BusinessConflictException`, `HttpStatus.FORBIDDEN` → Spring-Securitys `AccessDeniedException`. `ResponseStatusException` bleibt bewusst nur noch für die eine genuine 5xx-Fallback-Stelle im PDF-Generierungs-Pfad (`JobOrderHandoverReportService`) sowie für unbekannte Drittpartei-Stati im zentralen `handleResponseStatus`-Forwarder. `GlobalExceptionHandler` bekommt zwei neue Handler (`handleBadRequest`, `handleBusinessConflict`) plus den neuen stabilen Error-Code `BUSINESS_CONFLICT`. Die bisher hartcodierten Titel der `DuplicateEntity`- und `EntityInUse`-Handler werden ebenfalls über `MessageSource` lokalisiert (neue Schlüssel `problem.duplicate_entity.title/detail`, `problem.entity_in_use.title/detail`, `problem.business_conflict.title/detail`); die `HttpMessageNotReadableException`-Detail wird über den neuen Key `problem.unreadable_body.detail` aufgelöst, damit sich der generische `problem.bad_request.detail`-Fallback (für `BadRequestException` ohne Message) und der spezifische Parse-Error-Text nicht überschneiden. Acht Test-Dateien (`InventoryItemServiceBulkCheckoutTest`, `HangarImportServiceTest`, `InventoryItemServiceTest`, `JobOrderHandoverReportServiceTest`, `JobOrderHandoverServiceTest`, `JobOrderServiceTest`, `MissionSecurityServiceTest`, `SystemSettingServiceTest`, `JobOrderHandoverReportControllerTest`) sind auf die neuen Exception-Typen umgestellt; redundante `assertEquals(HttpStatus.X, ex.getStatusCode())`-Zeilen sind gestrichen, weil der Exception-Typ den Status nun selbst kodiert. Reproduzierbare Helper-Skripte unter `scripts/`: `replace_rse.py`, `fix_imports.py`, `fix_test_assertions.py`. Backend- und Frontend-Tests laufen vollständig grün.
- **Backend (i18n für RFC-7807-Fehlerantworten)**: Der `GlobalExceptionHandler` lokalisiert die `title`- und `detail`-Felder der Problem+JSON-Antworten jetzt über Spring's `MessageSource` anhand des `Accept-Language`-Headers (Default-Locale-Resolver: `AcceptHeaderLocaleResolver`). Bisher waren alle Strings hartcodiertes Englisch (Verstoß gegen die CLAUDE.md-Regel "every user-visible string comes from messages.properties"). Konkret: (1) Neue Resource-Bundles `backend/src/main/resources/messages{,_de,_en}.properties` mit 26 Keys nach Konvention `problem.<code>.title` / `problem.<code>.detail` (z. B. `problem.optimistic_lock.title`, `problem.access_denied.detail`). Der Default-Bundle (`messages.properties`) führt deutsche Texte als Fallback, das ist konsistent mit dem Frontend; `messages_en.properties` enthält die englischen Übersetzungen. Umlaute sind gemäß CLAUDE.md als `\uXXXX` codiert. (2) `GlobalExceptionHandler` injiziert nun `MessageSource` über `@RequiredArgsConstructor`; eine neue private `tr(String key, Object... args)`-Methode löst den Key über `LocaleContextHolder.getLocale()` auf und fällt — falls der Key fehlt — auf den Key selbst zurück (macht fehlende Übersetzungen in QA sichtbar, ohne Production zu crashen). Alle elf 4xx/5xx-Handler (`handleOptimisticLockingFailure`, `handlePessimisticLocking`, `handleAuthentication`, `handleAccessDenied`, `handleValidationExceptions`, `handleConstraintViolation`, `handleIllegalArgument`, `handleHttpMessageNotReadable`, `handleDataIntegrityViolation`, `handleTypeMismatch`, `handleMethodNotSupported`, `handleNotFound`, `handleAllExceptions`) wurden umgestellt; die parametrisierten Detail-Texte (`{0}` für Parameter-Namen, HTTP-Methoden) werden via Spring `MessageFormat` befüllt. (3) Die zwei Unit-Tests `GlobalExceptionHandlerTest` und `GlobalExceptionHandlerNotFoundTest` injizieren nun einen `ResourceBundleMessageSource` mit `setBasename("messages")` und setzen die Test-Locale auf `Locale.ENGLISH` (mit `@AfterEach LocaleContextHolder.resetLocaleContext()`), sodass die bestehenden Assertions auf die englischen Bundle-Werte unverändert greifen. Stable Error-Codes (`OPTIMISTIC_LOCK`, `ACCESS_DENIED`, …) bleiben unverändert; das Frontend liest weiterhin den `code` und bestimmt die Anzeigesprache.

### Fixed
- **Frontend (Toast-Fragment — Index-Seite warf 500 bei jedem Aufruf)**: Jeder Aufruf von `/` (oder anderen Seiten, die `fragments/toast` einbinden) brach die Thymeleaf-Renderung mit `SpelEvaluationException: EL1004E: Method matches(java.lang.String,java.lang.String) cannot be found on type org.thymeleaf.expression.Strings` ab. Ursache: `fragments/toast.html` rief in den Zweigen "General Success Toast (Param)" (Zeile 38) und "General Error Toast (Param)" (Zeile 67) `#strings.matches(param.success[0], '...')` bzw. `#strings.matches(param.error[0], '...')` auf. Die Thymeleaf-`Strings`-Utility besitzt jedoch keine `matches`-Methode — `matches` ist ein nativer SpEL-Infix-Operator. Fix: Umstellung auf den SpEL-Operator `param.success[0] matches '^[A-Za-z][A-Za-z0-9._-]{0,79}$'` bzw. `param.error[0] matches '...'`. Die sicherheitsrelevante Key-Pattern-Gate-Logik (verhindert beliebige Übersetzungs-Key-Auflösung via `?success=`/`?error=`) bleibt semantisch unverändert. Der Folgefehler `HttpMessageNotWritableException: No converter for [class java.util.LinkedHashMap] with preset Content-Type 'text/html;charset=UTF-8'` auf `/error` entfällt damit ebenfalls, da er nur als Kaskade der gescheiterten Toast-Renderung auftrat. Neuer Regressions-Test `HomeControllerMvcTest` rendert `index` end-to-end durch MockMvc (inkl. transitiv eingebundenem `fragments/toast`) und deckt sowohl `?error=`/`?success=` mit gültigem Pattern (Toast-Div wird emittiert) als auch ungültigem Pattern (Toast-Div fehlt, kein 500) ab, sodass ein Rückfall auf `#strings.matches(...)` sofort fehlschlägt.
